# План реализации проекта "Calendar Call Booking"

## Обзор проекта
Учебный проект Hexlet с API-спецификацией на TypeSpec для системы бронирования звонков в календаре. Требуется реализовать бэкенд и фронтенд согласно спецификации с минимальными изменениями.

## Цели
1. Реализовать полностью функциональный бэкенд API, соответствующий спецификации `main.tsp`
2. Создать интуитивный фронтенд для двух пользовательских ролей:
   - Владелец календаря (администратор)
   - Гость (пользователь, бронирующий слоты)
3. Обеспечить корректную работу бизнес-правил (отсутствие двойного бронирования)
4. Подготовить проект для демонстрации и тестирования

## Архитектура системы

### Выбранный технологический стек
- **Бэкенд**: Node.js + Fastify + TypeScript (≥6)
- **Фронтенд**: React 19 + TypeScript (≥6) + Vite 8 + Reatom v1000 (state management) + Mantine UI (компоненты)
- **База данных**: PostgreSQL
- **ORM**: Prisma (миграции, типизация) или Drizzle (альтернатива)
- **Валидация**: Zod (схемы валидации) + TypeBox (интеграция с Fastify)
- **Документация API**: OpenAPI 3.0 (генерируется из TypeSpec)
- **Сборка и инструменты**: pnpm (менеджер пакетов), oxlint (линтер), oxfmt (форматтер)

### Компоненты системы
```
┌─────────────────────────────┐     ┌─────────────────────────┐     ┌─────────────────┐
│        Фронтенд             │────▶│      Бэкенд API         │────▶│   База данных   │
│ (React 19 + Vite 8 + Mantine)│◀────│ (Fastify + TypeScript)  │◀────│   (PostgreSQL)  │
└─────────────────────────────┘     └─────────────────────────┘     └─────────────────┘
               │                                  │                             │
               │                                  │                             │
               ▼                                  ▼                             ▼
        Интерфейс пользователя           Бизнес-логика                 Хранение данных
        - Страница владельца            - Валидация (Zod)             - Таблицы:
        - Страница гостя                - Генерация слотов              • Owner
        - Календарь (Mantine Dates)     - Проверка конфликтов           • EventType
        - Управление состоянием (Reatom) - Управление бронями           • Slot
                                                                        • Booking
```

### Модели данных (соответствуют спецификации)
1. **Owner** - предопределённый владелец календаря
2. **EventType** - типы событий (название, описание, длительность)
3. **Slot** - временные слоты для бронирования
4. **Booking** - бронирования гостей

## Этапы реализации

### Этап 1: Подготовка и настройка проекта
1. Инициализация монорепозитория с pnpm workspaces (отдельные папки для бэкенда и фронтенда)
2. Настройка TypeScript ≥6 для бэкенда и фронтенда
3. Установка зависимостей: Fastify, React 19, Vite 8, Mantine, Reatom, Prisma, Zod, TypeBox
4. Настройка скриптов сборки и разработки с использованием pnpm
5. Генерация OpenAPI спецификации из TypeSpec (`tsp compile main.tsp --emit @typespec/openapi3`)
6. Настройка oxlint и oxfmt для линтинга и форматирования кода

### Этап 2: Разработка бэкенда
1. **Настройка базы данных и миграций**
   - Создание схемы Prisma по моделям TypeSpec
   - Написание миграций для создания таблиц (Owner, EventType, Slot, Booking)
   - Наполнение начальными данными (предопределённый владелец)

2. **Реализация CRUD для EventType**
   - GET /event-types - список типов событий (Fastify route)
   - GET /event-types/{id} - получение типа по ID
   - POST /event-types - создание типа (только владелец)
   - PUT /event-types/{id} - обновление типа
   - DELETE /event-types/{id} - удаление типа

3. **Реализация логики слотов (Slots)**
   - Генерация слотов на основе доступности (алгоритм с учётом длительности событий)
   - GET /slots - список доступных слотов с фильтрами (дата, тип события)
   - Бизнес-правило: предотвращение двойного бронирования (уникальные ограничения в БД)

4. **Реализация бронирований (Bookings)**
   - POST /public/bookings - создание бронирования гостем с проверкой конфликтов
   - GET /public/bookings/{id} - получение информации о бронировании
   - POST /public/bookings/{id}/cancel - отмена бронирования

5. **Реализация Owner API**
   - GET /owner/profile - информация о владельце
   - GET /owner/bookings - список предстоящих встреч с пагинацией и фильтрами

6. **Обработка ошибок и валидация**
   - Реализация error middleware в Fastify
   - Валидация входящих данных через Zod + TypeBox
   - Соответствие ошибок спецификации (NotFoundError, ValidationError, SlotConflictError)

### Этап 3: Разработка фронтенда
1. **Настройка проекта React**
   - Создание структуры компонентов с Feature-Sliced Design или аналогичной
   - Настройка маршрутизации (React Router)
   - Подключение Mantine UI (тема, провайдер, глобальные стили)

2. **Страница для гостя (публичная часть)**
   - Просмотр доступных типов событий (карточки с Mantine)
   - Выбор типа события и просмотр доступных слотов в календаре (@mantine/dates)
   - Форма бронирования с вводом данных гостя (React Hook Form + Zod)
   - Подтверждение бронирования с уведомлениями (Mantine notifications)

3. **Страница для владельца (административная часть)**
   - Управление типами событий (CRUD) с таблицей Mantine DataTable
   - Просмотр предстоящих встреч с фильтрами и пагинацией
   - Возможность отмены бронирований

4. **Компонент календаря**
   - Визуализация слотов по дням/неделям с использованием @mantine/dates
   - Выбор даты и времени
   - Подсветка занятых/свободных слотов (цветовая индикация)

5. **Интеграция с API**
   - Создание клиента API на основе сгенерированных типов из OpenAPI (или ручное описание)
   - Обработка состояний загрузки и ошибок через Reatom
   - Кэширование данных с помощью Reatom

### Этап 4: Интеграция и тестирование
1. **Интеграция фронтенда с бэкендом**
   - Настройка CORS в Fastify
   - Конфигурация прокси для разработки в Vite
   - Обработка аутентификации (если потребуется, но по спецификации не требуется)

2. **Тестирование**
   - Модульные тесты для бэкенда (Jest + Supertest)
   - Интеграционные тесты API (проверка endpoints)
   - E2E тесты для критических сценариев (Cypress или Playwright)
   - Тестирование бизнес-правил (конфликты бронирования)

3. **Валидация соответствия спецификации**
   - Сравнение реализованного API с OpenAPI спецификацией
   - Проверка всех endpoints на соответствие TypeSpec (статусы, схемы ответов)

### Этап 5: Деплой и документация
1. **Подготовка к деплою**
   - Docker-контейнеризация приложения (отдельные контейнеры для бэкенда, фронтенда, БД)
   - Настройка переменных окружения
   - Конфигурация для production сборки (Vite build, минификация)

2. **Документация**
   - Документация API (на основе OpenAPI, Swagger UI)
   - Инструкции по запуску проекта локально и в Docker
   - Описание архитектуры и принятых решений

3. **Демонстрация**
   - Создание демо-версии с примером данных
   - Подготовка скринкастов или скриншотов работы приложения

## Риски и решения

### Риск 1: Сложность генерации слотов
- **Решение**: Начать с простого подхода - фиксированные интервалы (например, каждый час). Затем усложнить до гибкой настройки.

### Риск 2: Конфликты бронирований
- **Решение**: Использовать транзакции в БД и уникальные ограничения на комбинацию slotId + startTime.

### Риск 3: Отсутствие аутентификации
- **Решение**: Следовать спецификации - система без аутентификации. Для административных endpoints можно использовать простой API-ключ в заголовках.

### Риск 4: Синхронизация фронтенда и бэкенда
- **Решение**: Использовать сгенерированные типы из OpenAPI спецификации для обеспечения типобезопасности.

## Критерии успеха
1. Все endpoints из спецификации реализованы и работают корректно
2. Фронтенд предоставляет полный пользовательский опыт для гостя и владельца
3. Бизнес-правило "нет двойного бронирования" соблюдается
4. Проект может быть запущен локально с минимальными усилиями
5. Код хорошо структурирован, документирован и покрыт тестами
